# Dummy Object Predicted Path Feature

## Overview

This feature enables dummy objects created in RViz to follow predicted trajectories from the perception system instead of simple straight-line motion. This provides more realistic object movement in simulations by leveraging the predicted paths generated by Autoware's perception pipeline.

## Quick Start

### What's New

- **RViz Tools**: Added "Predicted" checkbox to all dummy object placement tools (car, pedestrian, unknown)
- **Keyboard Shortcut**: Press 'P' to toggle prediction mode for objects
- **Action Type**: Objects created with prediction enabled use `action: PREDICT`

### Usage Example

```bash
# 1. Launch the dummy perception publisher
ros2 launch autoware_dummy_perception_publisher dummy_perception_publisher.launch.xml

# 2. In RViz, place objects with "Predicted" checkbox enabled

# 3. Ensure predicted objects are being published
ros2 topic echo /perception/object_recognition/objects
```

## Key Components

### 1. RViz Integration

All RViz dummy object tools (`car_pose`, `pedestrian_pose`, `unknown_pose`) now include:

- A "Predicted" checkbox property that can be toggled
- Support for the 'P' key to toggle objects to use predicted paths
- When enabled, objects are created with `action: PREDICT` instead of `action: ADD`

### 2. Dummy Perception Publisher Enhancements

The dummy perception publisher node subscribes to predicted objects and maintains a mapping between dummy objects and predicted objects based on Euclidean distance.

#### Core Functionality

- **Euclidean Distance Mapping**: Automatically assigns nearest predicted objects to dummy objects
- **Trajectory Interpolation**: Objects follow curved predicted paths instead of straight lines
- **Smart Remapping**: Handles disappeared predictions by finding new best matches
- **Validation System**: Ensures realistic movement with speed, position, and trajectory checks

#### Core Logic Flow

1. **Predicted Objects Reception**: The node subscribes to `/perception/object_recognition/objects` and maintains a buffer of recent predictions
2. **Mapping Assignment**: Uses Euclidean distance-based matching to assign predicted objects to dummy objects
3. **Position Interpolation**: Calculates object positions along predicted trajectories instead of straight lines
4. **Validation**: Ensures trajectory consistency and realistic movement

## Euclidean Distance-Based Mapping Algorithm

### Main Function: `updateDummyToPredictedMapping`

This function is called whenever new predicted objects are received. It manages the assignment of predicted objects to dummy objects.

#### Algorithm Steps

1. **Collect Available Predicted Objects**

   - Creates a set of all currently available predicted object UUIDs
   - Stores their current positions for distance calculations

2. **Find Disappeared Predicted Objects**

   - Identifies predicted objects that were previously mapped but are no longer available
   - Marks corresponding dummy objects for remapping to new predictions

3. **Update Dummy Object Positions**

   - Calculates current positions for all dummy objects
   - For objects with `action: PREDICT`, uses the mapped predicted trajectory
   - For others, uses straight-line motion

4. **Handle Remapping**

   - For dummy objects that lost their predicted object, finds new best matches
   - For unmapped dummy objects with `action: PREDICT`, finds initial matches

5. **Best Match Selection** (`findBestPredictedObjectMatch`)
   - Calculates Euclidean distance to all available predicted objects
   - Validates candidates using multiple criteria
   - Selects the closest valid predicted object

### Distance-Based Matching

- **Vehicles**: Max 2.0m initial matching distance
- **Pedestrians**: Max 3.0m initial matching distance
- Considers speed compatibility and trajectory similarity

### Position Calculation

When a dummy object has `action: PREDICT`:

- Uses `calculateTrajectoryBasedPosition` to interpolate along predicted path
- Falls back to extrapolation if beyond predicted trajectory
- Maintains smooth motion even during remapping

## Key Functions

### `calculateEuclideanDistance`

```cpp
double calculateEuclideanDistance(
  const geometry_msgs::msg::Point & pos1,
  const geometry_msgs::msg::Point & pos2)
```

Simple 3D Euclidean distance calculation between two points.

### `isValidRemappingCandidate`

Validates if a predicted object is suitable for remapping by checking:

- Distance threshold (vehicle: 2.0m, pedestrian: 3.0m)
- Yaw difference (vehicle: 15°, pedestrian: 45°)
- Speed compatibility
- Position difference from expected location
- Path length similarity
- Overall direction consistency

### `isTrajectoryValid`

Ensures smooth trajectory transitions by validating:

- Yaw change between predictions (max 90°)
- Path length consistency (max 3x change)
- Prevention of unrealistic direction changes

### `calculateTrajectoryBasedPosition`

Interpolates object position along the predicted trajectory:

1. Calculates elapsed time since prediction
2. Finds the appropriate path point based on time
3. Interpolates between path points for smooth motion
4. Falls back to extrapolation if beyond the predicted path

## Configuration Parameters

The feature uses ROS2 parameters with JSON schema validation. Key parameters in `config/dummy_perception_publisher.param.yaml`:

### Timing Parameters

- `predicted_path_delay`: 2.0 seconds - Delay before using predicted paths for new objects
- `min_keep_duration`: 3.0 seconds - Minimum time to maintain same prediction

### Vehicle-Specific Parameters

- `vehicle_max_remapping_distance`: 2.0m - Maximum distance for initial mapping
- `vehicle_max_remapping_yaw_diff`: 15° - Maximum yaw difference allowed
- `vehicle_max_speed_difference_ratio`: 1.05 - Maximum 5% speed difference
- `vehicle_min_speed_ratio`: 0.5 - Minimum 50% of dummy speed
- `vehicle_max_speed_ratio`: 1.5 - Maximum 150% of dummy speed
- `vehicle_speed_check_threshold`: 1.0 m/s - Speed validation threshold
- `vehicle_max_position_difference`: 1.5m - For path similarity checks
- `vehicle_max_path_length_ratio`: 1.1 - Maximum 10% path length difference
- `vehicle_max_overall_direction_diff`: 30° - Maximum direction difference

### Pedestrian-Specific Parameters

- `pedestrian_max_remapping_distance`: 3.0m - More lenient for unpredictable movement
- `pedestrian_max_remapping_yaw_diff`: 45° - Pedestrians can turn quickly
- `pedestrian_max_speed_difference_ratio`: 1.3 - Maximum 30% speed difference
- `pedestrian_min_speed_ratio`: 0.3 - Minimum 30% of dummy speed
- `pedestrian_max_speed_ratio`: 2.0 - Maximum 200% of dummy speed
- `pedestrian_speed_check_threshold`: 0.5 m/s - Lower threshold for walking
- `pedestrian_max_position_difference`: 2.5m - More variation allowed
- `pedestrian_max_path_length_ratio`: 1.5 - Maximum 50% path length difference
- `pedestrian_max_overall_direction_diff`: 60° - More direction change allowed

For detailed explanations of how each parameter is used, see [PARAMETER_EXPLANATION.md](PARAMETER_EXPLANATION.md)

## Validation Mechanisms

### 1. Speed Validation

- Compares dummy object speed with predicted object speed
- Ensures speed ratios are within acceptable ranges
- Different thresholds for vehicles vs pedestrians

### 2. Position Validation

- Checks if the current position matches expected position from trajectory
- Maximum allowed deviation varies by object type

### 3. Path Similarity

- Validates that predicted paths have similar lengths
- Ensures overall direction consistency
- Prevents sudden trajectory changes

### 4. Trajectory Continuity

- Maintains prediction stability for minimum duration
- Smooth transitions when remapping to new predictions
- Handles lost objects gracefully

## Random Path Selection for Pedestrians

When multiple predicted paths are available for a pedestrian, the system randomly selects one path to add natural variation to pedestrian movement patterns.

## Benefits

1. **Realistic Movement**: Objects follow curved paths and realistic trajectories
2. **Perception Integration**: Leverages existing perception predictions
3. **Smooth Transitions**: Handles prediction changes and lost objects gracefully
4. **Configurable**: Extensive parameters for different object types and scenarios
5. **Validation**: Multiple checks ensure realistic behavior

## Technical Details

### Message Types

- Input: `autoware_perception_msgs/PredictedObjects`
- Dummy Objects: `tier4_simulation_msgs/DummyObject` with `action: PREDICT`
- Output: Standard Autoware perception messages with trajectory-based positions

### Performance Considerations

- Maintains a buffer of last 50 predicted object messages
- Efficient Euclidean distance calculations for mapping
- Lazy evaluation of trajectory positions
- Configurable validation thresholds to balance accuracy vs performance
